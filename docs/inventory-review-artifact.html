Inventory: schema & competitive review

A read of Cargo's inventory data model and logic, benchmarked against Sortly, IDEA Yacht and Pinpoint Works — with the three things worth fixing first.

Scope 4-level nav · items / locations / movements schema Live data 349 items · 1 tenant Date 20/07/2026
Verdict

Cargo already has a richer item model and deeper yacht tooling than Sortly, but three foundations are unfinished: the location hierarchy is modelled three different ways, the stock-movement audit ledger isn't wired to manual changes, and low-stock / expiry alerting is schema-only and dormant. Close those and Cargo rivals IDEA on inventory while keeping its UX edge.

01

What we have today

The core inventory_items table is genuinely rich — 57 columns. In live data 341 items have barcodes and expiry dates and 340 have custom fields, yet only 6 use multi-location stock and none have reordering enabled: the advanced fields exist but are barely exercised.

Real strengths vs. the field

  • Import intelligence — Azure-PDF & spreadsheet import with hierarchy detection; PDF/XLSX export. Sortly is CSV-only.
  • Purchasing depth — supplier profiles, catalogue + price book, requisition→PO→receive. IDEA-tier.
  • Yacht-native — departments, command/chief/HOD visibility tiers, deck-plan locations, laundry & uniform types, alcohol + partial-bottle.
  • Bulk actions — bulk delete/move by id, drag-and-drop reordering.

What an item can carry

  • Quantity — total, per-location jsonb, partial-bottle fraction.
  • Units & pack maths — stocking unit, purchase unit, units-per-pack.
  • Cost — unit cost, purchase price, currency, value method.
  • Extensible — custom fields (GIN-indexed), tags, variants.
  • Identity — barcode, human-friendly Cargo ID, expiry date.
02

The structural finding: three location models

An item's location is modelled three different ways at once, and they aren't unified. No closure table — a string path, an adjacency tree and legacy columns living side by side. Rationalising this is the highest-leverage fix.

Active

Materialized-path strings

location + sub_location joined by  > . What the nav renders; moves rewrite descendants with LIKE cascades. Depth is actually unbounded.

Legacy

Denormalized L1–L4 columns

l1_id..l4_id / l1_name..l4_name on the item. "Kept for backward compat" — still read for category labels.

Physical

vessel_locations tree

A true adjacency list via parent_id with deck-plan coordinates. Items reference it only softly via default_location_id.

03

Gaps vs. table-stakes

Several expected features are present in schema but unfinished in practice — which is good news: the leverage is in wiring, not greenfield building.

AreaStateThe issue
Movement / audit trailLedger exists, not wired inventory_movements is a proper append-only ledger, but only the "receive delivery" flow writes to it. Manual +/-, bulk moves and deletes overwrite quantity with no ledger row.
Low-stock / par alertsSchema-only, dormant Three overlapping columns (par_level, reorder_point, restock_level); restock_enabled = 0 on every live row. No notifications.
Expiry alertsStored, not alerted 341/349 items have an expiry date but nothing warns on approach — critical for provisions, medical and safety gear.
Barcode scanningColumn only Barcodes are populated by import, but there's no scan-to-find / scan-to-count in the UI, and no label printing.
Inventory valuationFields, no roll-up Cost fields exist; value_method defaults to 'unknown'; no total-value report. Sortly ships this as standard.
Multi-location stockFragile Denormalized jsonb normalized across ~5 historical key spellings; only 6 items use it; no per-location ledger tie-in.

05

Recommended roadmap

The biggest wins are finishing what's already half-built — turning dormant schema into working features — before any greenfield work.

Now
Finish what's already half-built · high value, low new surface
1Wire the movement ledger into every stock change. Have quick +/-, bulk-move, edit and delete write inventory_movements rows. Turns an existing table into a full audit trail — the biggest credibility gap vs. IDEA.
2Activate low-stock + expiry alerts. Collapse the three par columns to one, surface a "needs attention" view, and notify. The data is already there (341 expiry dates) — it just isn't watched.
3Inventory valuation roll-up. Total value from unit_cost × quantity per folder/department. Cheap, and a standard expectation.
Next
Parity with Sortly's headline features
4Barcode scan-to-find / scan-to-count in the item UI (mobile camera), plus QR/label generation. We already store the barcodes.
5Rationalise the location model. Pick one source of truth, migrate the legacy L1–L4 columns, and make multi-location stock normalized rows rather than jsonb.
Later
Differentiation
6Deck-plan stock mapping — pin stock to cabins/compartments via vessel_locations coordinates. The Pinpoint GA idea, applied to inventory.
7Offline mobile for at-sea counting.
